# API Boundary Notes

- `GET /balance-allowance` currently returns `{ balance, allowances }`, where `allowances` is keyed by spender address. `clob-client` still models a single `allowance` string.
- `DELETE /cancel-market-orders` is documented in OpenAPI as requiring both `market` and `asset_id`, but `clob-client` examples use either field on its own. The SDK currently treats this request as a one-of filter and lets callers pass either or both.
- CLOB uses both `tokenId`/`token_id` and `assetId`/`asset_id` for the same CTF ERC1155 position identifier. Contract and order-entry shapes use `tokenId`, while older ledger, trade, and internal models still use `assetId`.
- `GET /prices-history` uses a `market` query parameter even when callers pass a CLOB token id rather than a condition id or Gamma market id.
- Authenticated CLOB notifications currently return `timestamp` as a string in live responses even though historical docs and models have treated it as a number.
- CLOB ID naming is inconsistent across legacy and unified surfaces. Legacy payloads still use forms like `orderID` and `tradeIDs`, while unified endpoints use `order_id` and `trade_ids`, and internal structs mix `Id` and `ID` spellings for the same identifier concepts.
- Order placement failures from reserved balance or allowance consumed by existing open orders are exposed only as free-form `ErrorMsg` text like `not enough balance / allowance`, sometimes with details such as `sum of active orders`. The API does not provide a stable structured reason that distinguishes wallet balance, allowance, open-order reservation, or matched-order reservation failures.
- `GET /auth/builder-api-key` is inconsistent across current references: legacy SDK consumers model it as builder key records with `key`, `createdAt`, and `revokedAt`, while the current auth docs describe a bare string array of active key ids. The SDK normalizes both shapes to builder key records.
- `relayer-v2` already serves newer unified account-transaction routes such as `/v1/account/transactions/*`, plus `POST /relayer/api/auth` and `POST /retry-deploy`, but the current public relayer OpenAPI still documents only the legacy transaction routes and read-only relayer key listing.
- Relayer nonce/submit params are advisory, not reserved. `GET /nonce` and `GET /v1/account/transactions/params` return the current optimistic nonce, but another submission can consume it before a caller signs and posts. Stale submissions then fail during execution, typically surfacing as a relayer transaction in `STATE_FAILED` rather than as a request-time validation error.
- Gamma stores `proxyWallet` on user and profile records, but Safe and proxy wallet addresses are also deterministic from the signer. If Gamma misses that write, public profile reads can omit the canonical wallet even though the SDK can still derive it locally.
- Token amount encoding is inconsistent across APIs. Some endpoints return raw smallest-denomination integers as decimal strings, others return unit-scaled decimal strings, and others use JSON numbers. For example, CLOB `GET /balance-allowance` returns raw on-chain balances and ERC20 allowances as `big.Int` strings, while other surfaces use decimal strings or numeric fields for token amounts.
- Pagination conventions differ materially across services. Gamma mixes offset envelopes like `{ data, pagination: { hasMore, totalResults } }` with keyset envelopes like `{ markets|events, next_cursor }`, while CLOB commonly uses `{ data, count, limit, next_cursor }` and some public APIs use bare arrays plus `limit`/`offset` query params.
- Gamma keyset payloads are not fully shape-compatible with the corresponding non-keyset payloads. For example, `/markets/keyset` may return arrays for `outcomes`, `outcomePrices`, and `clobTokenIds` where other Gamma endpoints return JSON-encoded strings, and `/events/keyset` nested related markets can omit a valid `conditionId`.
- Data position records expose `conditionId`, `asset`, and `eventId`, but not Polymarket `marketId`. Because Polymarket markets are effectively 1:1 with `conditionId`, callers can recover market identity indirectly, but the missing `marketId` makes portfolio-to-market workflows like redemption less direct than market-based flows elsewhere in the SDK.
- Data API `size` fields on positions, trades, and activity are normalized share quantities, not raw ERC1155/base-unit balances. The backend derives them as `balance / 1e6` for current market tokens and exposes them as floating-point JSON numbers, which is convenient for UX but weaker than an exact integer source for transaction-building flows.
- Balance/allowance refresh routes are currently inconsistent across CLOB surfaces. Backend source includes a unified `PUT /v1/account/balance-allowance`, public OpenAPI documents legacy `PUT /balance-allowance` plus `GET /balance-allowance/update`, `clob-client` uses `GET /balance-allowance/update`, and the deployed environment hit during SDK verification returned `404` for the unified route.
- Gamma currently exposes price-source hints for some crypto markets only indirectly via free-form `resolutionSource` / `resolution_source` URL strings. Short-window crypto up/down events often carry Chainlink stream URLs and hourly/daily price markets often carry Binance URLs, but other crypto market families return empty or null values. The app therefore infers websocket source choice from slug-derived market variants instead of relying on Gamma. This is problematic for API consumers because there is no stable dedicated field like `priceSource: 'chainlink' | 'binance' | 'pyth'`, so callers have to scrape URLs or replicate product-specific slug rules to choose the right realtime feed.
- CLOB websocket docs do not fully match the current subscriptions service output. In `clob` source tests, market `price_change` and `last_trade_price` events emit lowercase `side` values (`buy` / `sell`), user `trade` events emit `match_time` rather than the documented `matchtime`, and `trade.status` currently includes prefixed values such as `TRADE_STATUS_MATCHED` rather than the simplified documented status strings. SDK bindings therefore need to tolerate implementation-level payloads rather than relying on the public AsyncAPI alone.
- CLOB websocket `side` values are inconsistent across event families. Market-side events such as `price_change` and `last_trade_price` currently emit lowercase `buy` / `sell`, while user-side events such as `order`, `trade`, and nested `maker_orders` emit uppercase `BUY` / `SELL`. API consumers cannot treat `side` as a single stable enum across all CLOB websocket payloads without normalizing it themselves.
